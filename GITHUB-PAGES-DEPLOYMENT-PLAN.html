GitHub Pages Documentation Migration Plan - Technical Implementation

Overview

Migrating XMPro documentation from GitBook to GitHub Pages using DocFX-generated static sites with automated deployment pipelines.

Repository Structure

Source Documentation Repositories (Existing)

  • xmpro/public-docs-v4.4 → Has main branch (released version)
  • xmpro/public-docs-v4.5 → Exists but unreleased (nonprod only)
  • xmpro/public-docs-qa-v4.4 → Does not exist yet
  • xmpro/public-docs-qa-v4.5 → Does not exist yet

Target GitHub Pages Repositories (New)

Production: xmpro/public-documentation

├── 4.4/                    # Built from release/4.4 branch (future)
├── 4.5/                    # Built from main branch (when v4.5 released)
├── index.html              # Redirects to latest version
└── CNAME                   # documentation.xmpro.com

Non-Production: xmpro/public-documentation-nonprod

├── 4.4/                    # Built from public-docs-v4.4 main branch
├── 4.5/                    # Built from public-docs-v4.5 main branch
├── index.html              # Version selector page
└── CNAME                   # nonprod.documentation.xmpro.com

URL Mapping

Production (Released versions only):

  • documentation.xmpro.com → Latest released version
  • documentation.xmpro.com/4.4/ → Production v4.4 docs
  • documentation.xmpro.com/4.5/ → Production v4.5 docs (when released)

Non-Production (All versions):

  • nonprod.documentation.xmpro.com/4.4/ → QA v4.4 docs
  • nonprod.documentation.xmpro.com/4.5/ → QA v4.5 docs (unreleased)

GitHub Actions Pipeline Requirements

Branch-Based Production Deployment Strategy

Current State:

  • main branch → deploys to 4.4/ folder

Future State (when v4.5 is released):

  • main branch → deploys to 4.5/ folder
  • release/4.4 branch → deploys to 4.4/ folder

Production Deployment Logic

Pipeline needs branch detection logic:

- name: Determine deployment folder
  run: |
    if [[ "${{ github.ref }}" == "refs/heads/main" ]]; then
      # Currently main = 4.4, but will become 4.5 when released
      echo "DEPLOY_FOLDER=4.4" >> $GITHUB_ENV  # Change to 4.5 when ready
    elif [[ "${{ github.ref }}" == "refs/heads/release/4.4" ]]; then
      echo "DEPLOY_FOLDER=4.4" >> $GITHUB_ENV
    elif [[ "${{ github.ref }}" == "refs/heads/release/4.5" ]]; then
      echo "DEPLOY_FOLDER=4.5" >> $GITHUB_ENV
    fi

Pipeline Configuration Per Repository

xmpro/public-docs-v4.4:

  1. Non-Production Deployment:
    • Trigger: Push to main branch, completed PRs
    • Target: xmpro/public-documentation-nonprod folder 4.4/
  2. Production Deployment:
    • Trigger: Manual dispatch or release tags
    • Branch Logic: main → 4.4/ folder
    • Target: xmpro/public-documentation folder determined by branch

xmpro/public-docs-v4.5:

  1. Non-Production Deployment Only:
    • Trigger: Push to main branch, completed PRs
    • Target: xmpro/public-documentation-nonprod folder 4.5/
    • No production deployment until v4.5 is officially released

Release Transition Process

When v4.5 is ready for production release:

  1. Create release/4.4 branch in public-docs-v4.4
  2. Update production pipeline mapping: main branch → 4.5/ folder
  3. Enable production deployment for public-docs-v4.5
  4. Update production index.html to redirect to latest version

Build Process (Both Workflows)

  1. Triggers: Push to main (nonprod) or manual/release (prod)
  2. Build Steps:
    • Checkout source repository
    • Setup .NET SDK
    • Install DocFX CLI tool
    • Run docfx docfx.json to build static site
  3. Deployment Steps:
    • Use peaceiris/actions-gh-pages@v3 action
    • Determine target folder based on branch name
    • Deploy to external repository (prod or nonprod)
    • Use personal access token for cross-repo deployment

GitHub Organization Setup

  • Organization domain: xmpro.com (enables all subdomains)
  • Custom domains configured in repository settings:
    • xmpro/public-documentation → documentation.xmpro.com
    • xmpro/public-documentation-nonprod → nonprod.documentation.xmpro.com

DNS Configuration Required

documentation.xmpro.com CNAME xmpro.github.io
nonprod.documentation.xmpro.com CNAME xmpro.github.io

Security Requirements

  • Personal access token with repo permissions for cross-repository deployment
  • Token should be stored as repository secret DOCS_DEPLOY_TOKEN in each source repo

Current Implementation Priority

  1. Set up xmpro/public-documentation and xmpro/public-documentation-nonprod repositories
  2. Create branch-aware production deployment pipeline with current mapping (main → 4.4/)
  3. Create nonprod deployment pipelines for both repositories
  4. Test deployments and branch detection logic
  5. Verify DNS and domain configuration
  6. Plan transition strategy for v4.5 release

This plan provides flexible branch-based deployment that can evolve as new versions are released while maintaining clear separation between production and non-production environments.
